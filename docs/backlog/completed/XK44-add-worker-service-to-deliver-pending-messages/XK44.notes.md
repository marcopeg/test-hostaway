# Execution Notes - Add worker service to deliver pending messages
**Task**: ./XK44.task.md
**Plan**: ./XK44.plan.md

## Decisions

- Execute the accepted plan in the same turn as requested.
- Implemented the worker as a minimal JavaScript Node service with `pg`, avoiding a TypeScript build pipeline for this proof of concept.
- The worker drains currently pending rows one at a time on each 5-second poll.
- The worker uses a transaction for each insert/delete pair, but does not add row claiming, retry state, dead letters, or duplicate prevention.
- Docker Compose waits for `boot` to complete before starting the worker, and the worker also waits for Hasura health before polling.

## Problems Encountered

- The first local smoke run could not reach Hasura from inside the command sandbox. Re-ran `npm start` with approved elevated localhost access for the manual smoke test.
- The first subscription implementation selected multiple top-level Hasura fields, which broke the inbox at runtime with `subscriptions must select one top level field`.
- A subscription-only implementation could leave the direct thread route empty while waiting for the first WebSocket payload.

## Deviations From Plan

- Changed the inbox subscription to use a single `data_threads` root and derive visible stats from the subscribed thread payload instead of subscribing to separate aggregate roots.
- Kept an HTTP query for the initial inbox payload and added a single-root `data_threads` subscription through `subscribeToMore` for live updates. This prevents direct thread routes from rendering empty while the subscription connects.

## Known Limitations

- The worker assumes a single instance and happy-path delivery.
- The manual smoke inserted a test pending reply into the running local database and the worker processed existing pending seed rows as well.
