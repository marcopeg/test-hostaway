## Questions

### 1. Should the worker move pending messages through direct Postgres access or through Hasura GraphQL/admin APIs?

This affects dependencies, configuration, transaction handling, and whether the worker bypasses Hasura permissions.

Answer:

direct postgres access, but it can use the same connection string given to hasura. it's still a poc.

### 2. What should happen to `data.messages_pending` rows after successful processing?

Options include deleting the pending row after inserting into `data.messages`, adding a delivered/status column and updating it, or archiving delivery attempts in a new table.

Answer:

delete.

do not consider failures or retries.
this is a happy path simulation.

### 3. Should `apps/worker` run only as a local Node process, or should it also be integrated into Docker Compose and Makefile lifecycle targets?

For example, a minimal version could expose `npm run dev` under `apps/worker`; a more integrated version could add `make worker`, `make worker.install`, or a compose service.

Answer:

add a basic dockerfile so that it can be build and executed via docker-compose.
"docker compose up" should run the whole backend.
it should await for hasura healthcheck before starting.

### 4. What level of concurrency safety should this basic worker include?

The task asks to process messages in series, but implementation still needs to decide whether a single worker instance assumption is enough or whether row claiming/locking should prevent duplicates if two workers run accidentally.

Answer:

none. it should process 1 message at the time.
it's poc happy path.
it is used to demonstrate possible interactions.
this is not production.

### 5. What verification should be required before completing the task?

Possibilities include a manual smoke path, a script that inserts a pending message and checks it moves, a Vitest/integration test against local Hasura/Postgres, or some combination.

Answer:

you can try appending a reply for a random thread using your data model knowledge, wait out 6 seconds, and validate that it disappears from the pending and appears in the normal messages table.

no need to automate this test now.
