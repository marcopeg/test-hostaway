# Add worker service to deliver pending messages

## Elevator's Pitch

Create a basic Node.js worker at `apps/worker` that simulates the backend delivery process for operator replies by polling `data.messages_pending` and moving each pending row into `data.messages`. Update the inbox client to use subscriptions so other connected clients see thread and message changes as the worker processes replies.

## Business Gain

The inbox demo currently shows pending operator replies but has no backend process that finalizes them. A small worker makes the outbox flow more realistic and gives the product a concrete eventual-consistency path from pending reply to persisted message.

## Current State

- `data.messages_pending` exists and stores operator replies inserted through Hasura.
- `data.messages` is the canonical message table and requires exactly one of `operator_id` or `guest_id`.
- Seed data includes pending messages, and `INTERVIEW.md` calls out that the current optimistic outbox lacks a provider acknowledgment/delivery process.
- The repo currently has `apps/bff` and `apps/webapp`; there is no `apps/worker` service yet.
- The inbox currently uses a GraphQL query and calls `refetch()` after inserting a pending reply, which means another open client does not automatically see thread/message updates.
- The Apollo client already has WebSocket subscription support configured.

## Desired State

- A basic Node.js service lives under `apps/worker`.
- The worker connects directly to Postgres using the same database connection string available to Hasura/local backend services.
- The service polls for new rows in `data.messages_pending` every 5 seconds.
- Pending messages are processed one at a time in a deterministic order, such as oldest `sent_at` then `id`.
- For each pending row, the worker appends an equivalent operator message to `data.messages` and deletes the pending row so it no longer appears as pending.
- The worker is packaged with a basic Dockerfile and integrated into Docker Compose so `docker compose up` starts the backend stack with the worker.
- The worker waits for Hasura's health check before starting its polling loop.
- The inbox switches its thread/message data loading from query/refetch to subscriptions for the thread list and selected message list, so if client 1 replies, client 2 sees the pending and delivered updates.
- The behavior intentionally simulates happy-path delivery/backend processing without integrating a real channel provider.

## Definition of Success

- Running the worker alongside the existing Hasura/Postgres stack causes pending messages to become persisted messages without manual database intervention.
- Running `docker compose up` starts the backend stack and the worker after Hasura is healthy.
- Processing is serial and intentionally happy-path: the worker processes one pending row at a time and deletes it after inserting the persisted message.
- A second open inbox client receives updates through subscriptions when another client sends a reply and the worker delivers it.
- A manual smoke check can insert or submit a pending reply, wait at least 6 seconds, and confirm it disappears from `data.messages_pending` and appears in `data.messages`.

## Assumptions

- The worker connects directly to Postgres; this is acceptable because the task is a proof of concept.
- The worker does not need real provider delivery, retries, dead-letter queues, failure handling, concurrency protection, or a status state machine.
- The message should be copied with `thread_id`, `tenant_id`, `operator_id`, `message`, and `sent_at`; `guest_id` should be null for operator messages.
- The implementation assumes a single worker instance for the demo.

## Constraints

- Poll every 5 seconds.
- Process pending messages in series for simplicity.
- Keep the worker basic, happy-path, and demo-focused.
- Do not consider failures or retries.
- Do not handle conflicting or concurrent replies.
- Respect the existing Hasura/Postgres development workflow and use Makefile integration where appropriate.

## Acceptance Criteria

- `apps/worker` exists with the minimal Node.js project files needed to run the worker.
- `apps/worker` includes a basic Dockerfile.
- Docker Compose starts the worker as part of the backend stack.
- The worker waits for Hasura health before beginning its polling loop.
- The worker can be configured from the database connection string used by the local backend/Hasura environment without hardcoding new secrets.
- Every polling tick claims or selects pending rows and processes them one at a time.
- Each processed pending row creates a row in `data.messages` with matching tenant, thread, operator, message, and sent timestamp.
- Processed rows are deleted from `data.messages_pending`.
- The worker logs poll activity, successful deliveries, and row-level failures.
- The inbox thread list and message list use GraphQL subscriptions instead of query/refetch for live updates.
- When client 1 sends a reply, client 2 can see the pending reply and then the delivered persisted message without manually refreshing.
- Manual verification can append a reply for a known thread, wait at least 6 seconds, and validate that the row disappears from `data.messages_pending` and appears in `data.messages`.

## Dos

- Keep the implementation intentionally small.
- Prefer deterministic processing order.
- Make local configuration explicit.
- Use direct Postgres access.
- Delete the pending row after successful insert into `data.messages`.
- Add enough Docker Compose integration for `docker compose up` to run the worker with the backend.
- Update the inbox client to rely on subscriptions for cross-client thread/message updates.

## Don'ts

- Do not build a real provider integration.
- Do not add a broad delivery status machine unless refinement decides the schema needs it.
- Do not add retry, dead-letter, failure-state, or duplicate-prevention machinery.
- Do not automate the smoke test in this task.
- Do not handle conflicting or concurrent reply workflows.
