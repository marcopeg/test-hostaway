# Set up Playwright UI test suite

## Elevator's Pitch

Add the minimal Playwright browser test setup needed to exercise the React interface from the webapp package, expose it through make test.ui, and seed the suite with overview and inbox coverage.

## Business Gain

Browser-level UI tests give fast confidence that the main operator interface still opens, navigates, and renders seeded inbox data after frontend, GraphQL, or styling changes.

## Current State

- TESTING.md recommends Playwright for browser-level happy path and visual end-to-end checks around the inbox.
- The repository currently has no automated tests.
- apps/webapp contains the React frontend and is the package where frontend test dependencies and Playwright config should live.
- The documented local frontend URL is http://localhost:5173.
- The default local stack is prepared with make down, make start, and make init.
- The existing draft AD81 covers Vitest Hasura API query tests and explicitly excludes Playwright/browser E2E setup.

## Desired State

- Playwright is installed and configured for browser-level UI tests under apps/webapp.
- apps/webapp exposes npm scripts for running the Playwright suite, following TESTING.md guidance.
- The root Makefile exposes make test.ui to run the Playwright UI suite.
- The Playwright config starts the Vite frontend with webServer and can reuse an existing local dev server when available.
- The first UI test opens the app, navigates to overview, and validates that overview is marked as a work in progress.
- The second UI test navigates to inbox, validates that threads are visible, opens the first visible thread, and verifies the first rendered message is from a guest.

## Definition of Success

- After the local backend and seed data are prepared, running make test.ui executes the Playwright UI suite.
- The suite passes against the local frontend and default seeded data.
- The implemented scope remains focused on browser UI coverage; API-only Vitest work remains in AD81.

## Additional Context

- Use TESTING.md as the implementation guide, especially the Playwright: Browser E2E And Visual Checks section.
- Suggested config path from TESTING.md: apps/webapp/playwright.config.ts.
- Suggested test path from TESTING.md: apps/webapp/tests/e2e/inbox.spec.ts.
- Long-term selector hardening guidance in TESTING.md recommends role/label selectors first and data-testid attributes where labels are dynamic or duplicated.

## Assumptions

- make test.ui assumes the local backend is already running and initialized.
- Playwright's webServer starts the frontend dev server and reuses an existing server outside CI.
- The app has routable overview and inbox views that can be reached through either URLs or visible navigation controls.
- The seeded inbox contains at least one thread whose first displayed message can be identified as coming from a guest.
- If accessible labels are insufficient for stable selectors, small data-testid additions are acceptable.

## Constraints

- Keep Playwright files inside apps/webapp unless existing project conventions require a different location.
- Keep the first tests deterministic against the default seed data.
- Do not broaden this task into API-only Vitest coverage or mutation tests.
- Do not introduce broad visual snapshot coverage unless needed for the requested assertions.

## Acceptance Criteria

- Playwright test dependencies are added to apps/webapp.
- apps/webapp contains a Playwright config that runs tests from an e2e test directory.
- The Playwright config starts Vite on the documented local frontend port and reuses an existing server outside CI.
- apps/webapp/package.json includes a script for running the Playwright UI suite.
- The root Makefile includes a test.ui target that runs the Playwright UI suite from apps/webapp.
- A UI test opens the app, navigates to overview, and asserts the overview page shows the visible "Work in progress" state.
- A UI test navigates to inbox and asserts that one or more threads are rendered.
- The inbox UI test opens the first visible thread.
- The inbox UI test verifies the first rendered message in the opened thread is from a guest.
- Running make test.ui passes against the default local stack and frontend.

## Dos

- Follow TESTING.md where it matches this scope.
- Prefer user-facing role, text, and label selectors for stable UI interactions.
- Add small test ids only where the UI has repeated or ambiguous text.
- Keep Makefile naming consistent with the existing project targets.

## Don'ts

- Do not implement the Vitest Hasura API suite in this task.
- Do not require snapshot updates for normal make test.ui runs.
- Do not reset or reseed backend state inside make test.ui unless refinement explicitly chooses that lifecycle.
- Do not depend on arbitrary timeouts where Playwright assertions can wait for UI state.
