# Set up Vitest Hasura API query tests

## Elevator's Pitch

Add the minimal Vitest test environment needed to run Hasura GraphQL API query tests from the webapp package, then cover tenant-scoped thread/message visibility with one focused test.

## Business Gain

Automated API query tests make the Hasura permission model easier to verify after schema, metadata, seed, or frontend query changes. The first suite should prove tenant isolation without introducing browser E2E scope or mutation-side effects.

## Current State

- TESTING.md documents Vitest as the recommended tool for Hasura GraphQL API tests.
- The repository currently has no automated tests.
- apps/webapp/package.json has Vite, React, TypeScript, and npm scripts for dev, build, lint, and preview, but no Vitest dependency or test scripts.
- The root Makefile has webapp lifecycle targets but no test.api target.
- The default local Hasura endpoint is http://localhost:8080/v1/graphql with admin secret hasura.
- Seed data includes tenants gre, blu, and sun, with tenant-specific data_threads, data_messages, and data.messages_pending rows.
- Relevant seeded thread/message examples include gre thread Arrival time and door code with gre messages and blu thread Air conditioning stopped cooling with blu messages.
- Hasura metadata scopes data_threads, data_messages, and data_messages_pending reads by x-hasura-tenant-id for the user role.

## Desired State

- Vitest is installed and configured only as much as needed to run Hasura GraphQL API query tests from apps/webapp.
- apps/webapp exposes npm scripts for the Vitest suite, following the TESTING.md guidance.
- apps/webapp exposes the Hasura API suite through `npm run test:api`.
- A root Makefile target named `test.api` delegates to the webapp `test:api` script.
- The API test helper sends normal application Hasura headers for role, tenant id, and user id.
- Tests call Hasura directly at HASURA_GRAPHQL_URL or the documented local default.
- The first suite contains one test that pulls threads with nested persisted `messages` for two existing tenants and one unknown tenant.
- The first test verifies two existing tenants receive different message sets.
- The first test verifies each returned thread has at least one persisted message.
- The first test verifies a non-existing tenant receives no threads and no messages.
- The suite resets backend state between tests by running `make migrate.rebuild && make seed`, either through the Makefile commands or by directly using the equivalent Hasura APIs.

## Definition of Success

- After the local backend is already running, running `make test.api` executes the Vitest Hasura query suite.
- The suite passes against the seeded local Hasura stack.
- The implemented scope is limited to query tests against Hasura; Playwright, browser E2E tests, and mutation tests are not introduced in this task.
- Test state is rebuilt and seeded between tests through the existing migration and seed workflow.

## Assumptions

- The Hasura/Postgres stack is already running before `make test.api` is executed.
- `make test.api` does not need to start or stop Docker services.
- The test suite may invoke `make migrate.rebuild && make seed` between tests to restore deterministic state.
- Using the Hasura admin secret together with x-hasura-role: user is acceptable for local permission tests, matching TESTING.md.
- The unknown tenant can use any id that is not seeded, such as xxx or missing-tenant.
- The test should compare observable thread/message contents rather than depending only on row counts.

## Constraints

- Prepare only the Vitest environment for Hasura query tests.
- Do not add Playwright or browser E2E setup in this task.
- Do not add mutation tests in this task.
- Keep the suite deterministic against the default seed data.
- Use environment variables for HASURA_GRAPHQL_URL and HASURA_ADMIN_SECRET with documented local defaults.
- Run tests serially if needed to keep Hasura API tests predictable.
- Query only persisted `messages` in the first test; do not include `pending_messages` in this task.

## Acceptance Criteria

- Vitest is added as a dev dependency in apps/webapp.
- apps/webapp/package.json includes a `test:api` script that runs Vitest for the Hasura API tests.
- The root Makefile includes a test.api target that runs the Hasura API test suite from apps/webapp.
- A typed GraphQL helper exists for Hasura API tests and sends content-type, x-hasura-admin-secret, x-hasura-role, x-hasura-tenant-id, and x-hasura-user-id headers.
- A Hasura query test file exists under apps/webapp/tests/hasura.
- The suite contains one test case that queries data_threads with nested messages for tenant gre and tenant blu.
- The test queries only persisted `messages`, not `pending_messages`.
- The test asserts both existing tenants receive at least one thread.
- The test asserts every returned thread has at least one persisted message.
- The test asserts both existing tenants receive at least one persisted message.
- The test asserts the two existing tenants receive different message sets or thread/message content.
- The test asserts all returned thread and message tenant_id values match the requested tenant.
- The same test queries a non-existing tenant and asserts it receives no threads and no messages.
- The suite resets state between tests using `make migrate.rebuild && make seed` or the equivalent Hasura API calls.
- Running make test.api passes against the default local stack when the backend is already running.

## Dos

- Follow the TESTING.md Vitest guidance where it matches this narrower scope.
- Keep test utilities small and local to apps/webapp/tests/hasura.
- Prefer explicit seeded tenant ids and seeded message content for readable assertions.
- Keep the Makefile target simple and aligned with existing app task style.
- Prefer `npm run test:api` inside apps/webapp and have root `make test.api` delegate to it.

## Don'ts

- Do not add Playwright dependencies, config, snapshots, or E2E tests.
- Do not test pending-message insertion or cross-tenant mutation rejection in this task.
- Do not make make test.api start or stop Docker services.
- Do not include `pending_messages` in the first query test.
- Do not use Hasura admin-only GraphQL behavior for the assertions; exercise the user role permission path.
