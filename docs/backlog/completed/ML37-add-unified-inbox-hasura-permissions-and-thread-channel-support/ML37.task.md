# Add unified inbox Hasura permissions and thread channel support

## Elevator's Pitch

Update the Hasura metadata, migrations, and seeds so the unified inbox GraphQL API exposes tenant-scoped user permissions for messaging subscriptions and models each thread's channel explicitly.

## Business Gain

The React Apollo client needs a usable subscription surface for the Hostaway messaging app. Tenant-scoped permissions and explicit thread channel data let the UI subscribe to inbox threads, single-thread conversations, pending messages, listing details, and booking details without exposing cross-tenant data.

## Current State

The unified inbox data model exists in `QC03`, with tables for tenants, users, guests, channels, listings, bookings, threads, messages, and pending messages. Current metadata and seeds do not yet define the requested basic `user` role permissions for the messaging app. `data.threads` is associated with listings and bookings, but the user now wants each thread to reference its channel explicitly so inbox subscriptions can read `channel_id` and channel logo directly from the thread.

## Desired State

- The Hasura metadata defines basic select and subscription-ready permissions for role `user` using the Apollo session variables `x-hasura-role: user`, `x-hasura-tenant-id: xx`, and `x-hasura-user-id: yy`.
- `data.threads` has an explicit `channel_id` foreign key to `data.channels`; the migration, relationships, indexes, and seeds are updated accordingly.
- `data.threads.channel_id` is mandatory for every thread, including listing-only threads.
- The thread-list subscription can return each thread's `id`, `title`, `mood_value`, `topic_value`, `channel_id`, and related channel logo/icon.
- The thread-list subscription can return the latest message information through the existing messages relationship ordered by `sent_at` descending with `limit: 1`, including operator id/name, guest id/name, and `sent_at`.
- A single-thread subscription can return the full message list, full pending-message list, related listing details, and related booking details.
- Read and subscription permissions are tenant-scoped with `x-hasura-tenant-id` wherever the table owns tenant data.
- `x-hasura-user-id` is used for inserting pending outgoing messages by presetting `data.messages_pending.operator_id`; it is not used to restrict tenant reads in this first pass.
- The only write permission in this task is `user` role insert permission on `data.messages_pending`, with `tenant_id` preset from `x-hasura-tenant-id`, `operator_id` preset from `x-hasura-user-id`, and validation that the referenced thread belongs to the same tenant where Hasura metadata can express that.
- Relationships needed by these subscriptions are present in metadata, including thread-to-channel, thread-to-listing, thread-to-booking, thread-to-messages, thread-to-pending-messages, message-to-operator, message-to-guest, and booking/listing/channel relationships.

## Definition of Success

- A migration adds `data.threads.channel_id`, its foreign key to `data.channels`, and practical indexes for tenant/channel subscription filters.
- Seeded threads include valid channel references that are coherent with related bookings where a booking exists.
- Hasura metadata tracks the new relationship and remains consistent.
- The `user` role can subscribe to tenant-scoped thread lists with channel details and last-message author details.
- The `user` role can subscribe to a tenant-scoped single thread with all messages, pending messages, listing details, and booking details.
- The `user` role can insert pending outgoing messages only into `data.messages_pending`, with tenant and operator values derived from session variables rather than client-provided values.
- The implementation can be validated through the existing make-based Hasura lifecycle and at least one GraphQL smoke query using the stated session headers.

## Additional Context

This is a follow-up to `QC03: Define unified inbox data model`. The request is specifically about Hasura metadata permissions for the messaging app and the schema/seed refactor required to associate each thread with a channel explicitly.

## Assumptions

- The Hasura BFF will receive the role and tenant/user IDs as session variables from Apollo Client.
- The first permission pass is mostly read/subscription focused, with one explicit insert path for optimistic outgoing messages in `data.messages_pending`.
- Last-message data should be exposed through the existing messages relationship with ordering and limit for the MVP rather than through a dedicated view or relationship.
- Channel logo means the existing channel icon/logo field in `data.channels`, unless implementation reveals a different column name.

## Constraints

- Keep reads tenant-scoped and avoid cross-tenant reads.
- Do not restrict read permissions by `x-hasura-user-id` in this task.
- Restrict writes to inserting rows into `data.messages_pending`; do not add update/delete permissions or writes to persisted messages.
- Preset `tenant_id` and `operator_id` for pending-message inserts from session variables instead of trusting client input.
- Do not add denormalized unread or inbox status fields unless needed to satisfy the subscription surface.
- Preserve the make-based Hasura migration, metadata, and seed workflow.
- Keep this task focused on backend schema, metadata, permissions, seeds, and smoke validation rather than frontend Apollo integration.

## Acceptance Criteria

- Running the Hasura metadata consistency check reports consistent metadata.
- Running migrations and seeds from a clean database succeeds.
- `data.threads` includes non-null `channel_id` with a foreign key and tracked object relationship to `data.channels`.
- All seeded threads have a valid `channel_id`; threads with bookings use a channel value that does not contradict the booking channel.
- Role `user` has select permissions needed for tenant-scoped subscriptions on threads, channels, messages, pending messages, listings, bookings, guests, and users/operators.
- Role `user` has insert permission on `data.messages_pending` only, with `tenant_id` preset from `x-hasura-tenant-id`, `operator_id` preset from `x-hasura-user-id`, and thread tenant validation where supported by Hasura metadata.
- A smoke subscription/query with headers `x-hasura-role: user`, `x-hasura-tenant-id`, and `x-hasura-user-id` can retrieve the requested thread-list fields.
- A smoke subscription/query with the same headers can retrieve a single thread with full messages, full pending messages, listing details, and booking details.
- A smoke mutation with the same headers can insert a pending outgoing message without supplying trusted tenant or operator values from the client.

## Dos

- Use Hasura metadata YAML patterns already present in the repository.
- Update migrations and seeds together so metadata can apply cleanly.
- Prefer explicit relationships and permission filters over frontend-side filtering.
- Use the messages relationship with `order_by: {sent_at: desc}` and `limit: 1` for MVP last-message retrieval.
- Document any GraphQL shape constraints discovered during implementation.

## Don'ts

- Do not leave the new `channel_id` nullable in seeded data if the subscription requires every thread to show a channel.
- Do not grant broad unfiltered access to tenant-owned tables.
- Do not add a dedicated `last_message` view or denormalized last-message fields for this MVP unless Hasura relationship ordering cannot satisfy the subscription.
- Do not let clients choose trusted `tenant_id` or `operator_id` values for pending-message inserts.
- Do not implement unrelated frontend UI work in this task.
