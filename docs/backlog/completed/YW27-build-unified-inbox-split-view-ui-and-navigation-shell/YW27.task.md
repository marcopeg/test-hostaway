# Build unified inbox split-view UI and navigation shell

## Elevator's Pitch

Replace the mocked todo-list demo with a real Hostaway-style inbox experience. Add the main and top navigation shell in BaseLayout, route real menu items to work-in-progress pages, register /inbox as the implemented route, and build a reusable split layout for thread list, message conversation, and optional details panel.

## Business Gain

The operator needs first-glance situational awareness for guest messaging. A proper inbox with channel icons, unread-style emphasis, message context, booking/listing details, and aggregated mood/topic/stage signals makes it possible to quickly identify which conversations need attention and why.

## Current State

The webapp currently routes / to a todo-list container and /$todoId to todo details. BaseLayout has a static Hostaway-like side menu whose links all point to /. The existing apps/webapp/src/containers/todos-list/TodosListUI.tsx already contains a mocked three-column conversation layout, but it is still driven by todo data and should be removed as demo code. A related backend WIP task, ML37, is adding unified inbox Hasura permissions, tenant-scoped thread/message access, channel support, listing details, booking details, and pending outgoing message inserts.

## Desired State

- BaseLayout contains the main menu and top menu for the application shell.
- Main menu entries perform real navigation.
- Every main menu route is registered and renders a Work in progress page, except /inbox.
- The root route `/` redirects to `/inbox` after the todo demo is removed.
- /inbox renders the real inbox view.
- The accepted main menu paths are `/overview`, `/calendar`, `/reservations`, `/smart-locks`, `/inbox`, `/tasks`, `/reviews`, `/guest-payments`, `/financial-reporting`, `/owner-statements`, `/listings`, `/channel-manager`, and `/settings`.
- Direct import routes include a new views/inbox route module.
- The todo-list/todo-details demo experience is removed from the app routes and UI surface.
- A reusable components/layout/split-layout/SplitLayout component supports a left list column, center content column, and optional right details panel.
- SplitLayout uses the same background for the left and right areas, while the center panel uses the main content background, top border corners, and a small top margin from the layout.
- Each SplitLayout area scrolls independently when its content grows.
- views/inbox uses SplitLayout.
- views/inbox/containers/thread-list is specific to the inbox view and renders the left column.
- /inbox with no selected thread shows a stats/dashboard panel in the center area.
- Clicking a thread navigates to `/inbox/$threadId` and opens messages in the center plus details on the right.
- Thread rows show the channel icon, guest/listing context, last-message preview, mood/topic/stage indicators where useful, and bold unread-like emphasis when the last message was sent by the guest/user rather than an operator.
- Thread unanswered/unread-style state is computed from the logically merged timeline of persisted `messages` and outgoing `messages_pending`; a pending outgoing operator message can make a previously guest-latest thread no longer count as unanswered.
- Messages are visually distinct for guest versus operator senders in a professional chat-like layout.
- Messages show sender name, sender icon, short time labels, and full time expansion on hover.
- Messages in the selected-thread view include both persisted `messages` and related `messages_pending`, merged and sorted into one conversation timeline.
- The message list has a sticky compact Reply affordance at the bottom that expands to a textarea and submit action. If the scroll is already at the end, the reply control is simply visible in context.
- The details panel shows mood and topic indicators, listing information first, a mocked listing image carousel, proposed listing fields that may be added later, booking timing, a clear from-to date visualization, and booking stage.
- Mood, booking stage, and topic labels are represented by polished reusable visual components optimized for quick scanning.
- The dashboard may include additional useful situational-awareness metrics beyond the required totals and aggregations when the available Hasura data supports them.

## Definition of Success

- Navigation no longer points all menu items to /, and active state follows the current route.
- The /inbox route is registered and renders the inbox split view.
- Non-inbox menu routes render a consistent Work in progress page.
- The old todo-list/todo-details demo UI is removed from the routed application surface.
- The inbox dashboard shows total messages, total unanswered, aggregated mood counts, aggregated booking-stage counts from related bookings, and aggregated topic counts.
- Total unanswered reflects the latest logical message per thread after merging persisted messages with pending outgoing messages.
- The thread list shows real channel icons from the inbox data surface.
- Selecting a thread changes the URL and displays the message list and details panel.
- The split layout remains usable at realistic desktop sizes with independent scrolling in each column.
- The UI communicates first-glance situational awareness through density, hierarchy, and label treatments rather than decorative filler.
- If Hasura permissions do not expose the required aggregate fields, the task updates metadata where appropriate and records concrete limitations in chat and task notes. If the data model itself cannot represent a requested metric, the task records a concrete backend/data-model proposal rather than making broad schema changes.

## Additional Context

Prefer the existing Hostaway-like visual language already present in BaseLayout and TodosListUI, but convert it from todo/demo-driven UI into real inbox-oriented components and routes. The backend/data surface is related to ML37, which establishes tenant-scoped inbox permissions, thread channel support, messages, pending messages, listings, bookings, and channel relationships.

## Assumptions

- The initial inbox can use the existing unified inbox Hasura data shape from ML37 where available.
- Mocked listing carousel images and proposed listing fields are acceptable until the data model is extended.
- `/inbox/$threadId` is the selected-thread route shape.
- Total unanswered can be derived from existing message/thread data for the first pass by merging persisted messages with pending outgoing messages and inspecting the latest logical sender, unless implementation discovers the data model needs an explicit unread/unanswered field.
- Topic, mood, and booking stage values already exist or can be read from related thread/booking fields exposed by Hasura.

## Constraints

- Keep this task focused on the frontend inbox experience and the minimal Hasura permission adjustments needed for aggregations.
- Do not make broad data model changes during this task; propose them to the developer if discovered.
- Metadata changes that enable required Hasura aggregate queries for the existing model are in scope.
- Do not keep todo-list demo abstractions in the final routed UI.
- Do not create a marketing landing page; the app should open into operational product UI.
- Keep layout density suitable for repeated property-management work and first-glance scanning.

## Acceptance Criteria

- BaseLayout includes a functional main menu and top menu consistent with the Hostaway-like operational UI style.
- Each main menu item uses router navigation and has a registered route.
- `/` redirects to `/inbox`.
- The main menu paths are `/overview`, `/calendar`, `/reservations`, `/smart-locks`, `/inbox`, `/tasks`, `/reviews`, `/guest-payments`, `/financial-reporting`, `/owner-statements`, `/listings`, `/channel-manager`, and `/settings`.
- All registered menu routes except /inbox display a Work in progress page.
- /inbox is implemented as a view under apps/webapp/src/views/inbox and added to the direct import route list.
- A reusable apps/webapp/src/components/layout/split-layout/SplitLayout component exists and is used by the inbox view.
- SplitLayout supports left, center, and optional right regions with independent scrolling and the requested background/content-panel treatment.
- The existing mocked TodosListUI demo is removed from the routed application and obsolete todo containers are cleaned up where no longer needed.
- views/inbox/containers/thread-list renders the thread list in the left column and remains inbox-specific.
- The inbox index state renders a stats container at views/inbox/containers/stats with total messages, total unanswered, mood counts, booking-stage counts, and topic counts.
- Thread rows show each thread's channel icon and use bold unread-style treatment when the latest logical message, after merging persisted and pending messages, is guest-sent.
- Clicking a thread navigates to `/inbox/$threadId` and renders center messages plus right details.
- The selected-thread message timeline merges persisted `messages` with related `messages_pending` and sorts them chronologically.
- Message bubbles distinguish operator and guest, show sender name and icon, and expose concise time labels with full timestamp on hover.
- The reply composer starts as a sticky compact control and expands to a textarea with submit.
- The details panel shows mood/topic indicators, listing-first details, a mocked image carousel, proposed listing fields, booking time, from-to dates, and booking stage.
- Mood, stage, and topic visual components are reusable and visually clear across dashboard, thread list, and details panel.
- Aggregation and thread-detail GraphQL queries run under the intended user role. If permissions or schema block the needed aggregations, the implementation includes the smallest appropriate Hasura metadata adjustment or a documented proposal rather than silently faking backend capability.
- Any aggregation or data-model limitations discovered during implementation are reported in chat and in task notes.

## Dos

- Reuse existing React, TanStack Router, Apollo/GraphQL, Tailwind, and local component patterns.
- Use direct import routes consistently with the app's routing setup.
- Keep inbox-specific containers under views/inbox/containers.
- Build reusable label/badge components for mood, stage, and topic.
- Validate responsive behavior and scrolling, especially around the three-column split layout and sticky reply composer.
- Prefer real GraphQL data over local mock data where the backend surface exists.
- Use Hasura aggregate queries where available, and adjust metadata narrowly when required to expose existing aggregate data.
- Add extra dashboard signals when they strengthen first-glance situational awareness and can be computed from available data.

## Don'ts

- Do not leave / as the only working navigation target.
- Do not preserve TodosListUI as a todo-driven inbox mock.
- Do not fake aggregation results if Hasura blocks the query; either adjust permissions or document the exact backend gap.
- Do not hide the selected thread details behind a modal for the desktop split-view MVP.
- Do not over-decorate the UI at the expense of operational readability.
