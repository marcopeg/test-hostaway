# Execution Notes - Define unified inbox data model
**Task**: ./QC03.task.md
**Plan**: ./QC03.plan.md

## Decisions

- Use one replacement Hasura migration for the unified inbox schema.
- Use plain `text` fields for booking stage, thread mood, and thread topic.
- Use `updated_at` on domain tables and omit `created_at` as requested.
- Keep pending messages minimal for the POC.
- Seed channel names are Airbnb, Booking.com, Vrbo, Expedia, and Google Vacation Rentals based on Hostaway channel manager/integration references.

## Problems Encountered

- Initial `make start` failed because the existing local Docker volume still contained the old todo schema (`auth.users` already existed). Resetting with `make down` and then running `make start` validated the replacement initial migration from a clean database.
- The repository has no `.env` file, so the Makefile printed its existing warning and used default Docker Compose environment values.

## Known Limitations

- The frontend still contains todo-oriented queries and routes; this task intentionally changed only the backend data model, metadata, and seed surface.
- Inbox ordering and unhandled state are intentionally derived from message data for this POC rather than denormalized on `data.threads`.

## Validation

- `make down`: reset the local Docker stack and database volume.
- `make start`: applied SQL migrations, Hasura metadata, and SQL seeds successfully from a clean volume.
- `make migrate.status`: reported `20260521161000 unified_inbox_data_model` as present in source and database.
- `make hasura.debug`: reported metadata is consistent.
- Postgres row counts: 3 tenants, 3 users, 12 guests, 5 channels, 15 listings, 8 bookings, 10 threads, 17 messages, and 2 pending messages.
- GraphQL smoke query returned `data_threads` with tenant, listing, booking/channel, messages, guest/operator, and pending message relationships.
