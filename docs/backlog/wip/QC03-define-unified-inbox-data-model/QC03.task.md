# Define unified inbox data model

## Elevator's Pitch

Define the PostgreSQL and Hasura data model for Hostaway unified inbox management, replacing the existing todo boilerplate schema with domain tables and relationships for inbox conversations and messages.

## Business Gain

A clear unified inbox schema gives the BFF a stable GraphQL surface for future inbox workflows, subscriptions, filtering, and authorization rules.

## Current State

The repository still contains the todo boilerplate data model under `apps/bff/migrations/default/20260521133000_basic_todos_app`, with matching todo metadata and seeds.

## Desired State

New Hasura migration(s) and seeds for the Hostaway Unified Inbox.

We will always work with tables in custom schemas, I will use the `schema.table` to refer to a table.
Add `updated_at` to domain tables. Do not add `created_at` for this iteration; upstream services update these read-ready tables and `updated_at` captures the last materialized event time.

Understand that this is the internal data model.
Each channel has an adapter service that mediates between the external channel that owns the data, and the internal representation of the data itself.

### auth.tenants

Contains Hostaway's customers.

- id: 3 letters so that it's easy to produce static seeds
- name: full name, something cool like "Grayhound Estates".
- icon: generate initials using the icon service
- updated_at

Seed: generate 3 tenants

### auth.users

- id: 3 letters so that it's easy to produce static seeds
- name: classic stuff like "John Doe"
- icon: generate initials using the icon service
- updated_at

Seed: generate 3 users

### data.guests

- id: 3 letters so that it's easy to produce static seeds
- tenant_id: reference to the tenant
- name: classic stuff like "Jane Doe"
- icon: generate initials using the icon service
- updated_at

Seed: generate enough guests to fill the seeded threads and bookings

### data.channels

- id: 3 letters so that it's easy to produce static seeds
- name: stick to real names like "AirBnb", "Bookings.com"
- icon: see if you can find a real url to the real logo
- updated_at

Seed: generate the 5 major channels, search online - best if you find that Hostaway integrates with it

### data.listings

- id: 3 letters so that it's easy to produce static seeds
- tenant_id: reference to the tenants
- name: make up cool names for holiday places
- updated_at

Seed: generate at least 15 listings associate with different tenants

### data.bookings

- id: 3 letters so that it's easy to produce static seeds
- tenant_id: reference to the tenants
- listing_id: reference to the internal representation of the listing
- channel_id: reference to the channel that owns the booking
- start_date: datetime, when the booking begins
- end_date: datetime, when the booking ends
- stage: plain text booking phase. Known values for this iteration are "pre-booking", "booked", "checked-in", and "checked-out". It will be reflected on the UI.
- updated_at

Seed: create enough bookings to fill the threads

### data.threads

- id: uuid
- tenant_id: reference to the tenants
- listing_id: referece to a listing (nullable)
- booking_id: reference to a booking (nullable)
- title: title of the thread, usually the first message's title
- mood_value: plain text. An external AI service will check the humor of the conversation. Known values for this iteration are "neutral", "angry", "frustrated", "litigious", and "happy".
- mood_updated_at: date time, last time thr ai reviewed the conversation
- topic_value: plain text. An external AI service will check the topic. Known values for this iteration are "check-in-issue", "guest-incident", "appliance-incident", "guest_complaint", and "payment_issue".
- topic_updated_at: date time, last time thr ai reviewed the conversation
- updated_at

For this POC, inbox ordering and unhandled state should be derived from `data.messages` via joins rather than denormalized on `data.threads`. Do not add `last_message_at`, `last_message_preview`, `last_message_direction`, `is_unhandled`, `assigned_user_id`, or `status` yet.

Seed: seed at least

- 5 threads with both listing and booking associated
- 5 threads with only listing associated

Use the mood flags and the topics freely, but leave some without

### data.messages

- id: uuid
- thread_id
- tenant_id
- operator_id: (nullable, this will flag a message from the operator)
- guest_id: nullable reference to the guest. A message may come from an operator or a guest; store guest authors explicitly because bookings can involve multiple guests.
- message: text, it's the body of the message
- sent_at: date time
- updated_at

For this iteration, `operator_id` and `guest_id` identify the message direction. Exactly one of them should be set for a normal persisted message.

Seed: enough messages for each thread so to display a proper conversation. Leave some threads only with the first guest message that means it's an unhangled thread

### data.messages_pending

This table is for outgoing messages that will have to go through the outbox pattern.
BFF will write to this table as a form of optimistic update.

- id: uuid
- thread_id
- tenant_id
- operator_id: (not nullable, this is only for operators)
- message: text, it's the body of the message
- sent_at: date time
- updated_at

Do not overbuild the outbox in this task. Keep `data.messages_pending` as the optimistic outgoing-message draft table and do not add status, attempts, last_error, or processed_at yet.

### Indexes and relationships

- Add primary keys for every table.
- Add foreign keys for tenant, user/operator, guest, channel, listing, booking, and thread references.
- Add practical indexes for tenant scoping and joins: tenant references, `data.bookings.listing_id`, `data.bookings.channel_id`, `data.threads.listing_id`, `data.threads.booking_id`, `data.messages.thread_id`, `data.messages.guest_id`, `data.messages.operator_id`, and `data.messages_pending.thread_id`.
- Add message ordering indexes for conversations, especially `data.messages(thread_id, sent_at)` and `data.messages_pending(thread_id, sent_at)`.

## Definition of Success

- The todo boilerplate migration is removed or replaced by domain-specific migration(s).
- The unified inbox data model is represented by Hasura-compatible SQL migrations.
- Static seed data exists for tenants, users, guests, channels, listings, bookings, threads, messages, and pending messages.
- Todo boilerplate metadata and seed artifacts are deleted or replaced without preserving the old todo model.

## Assumptions

- Hostaway unified inbox data needs to support conversations and messages at minimum.
- The BFF remains Hasura-backed and migration-driven through the existing `make` lifecycle.
- The first iteration optimizes for a POC and can rely on joins for inbox ordering and unhandled state.

## Constraints

- Do not keep the todo boilerplate schema as active product schema.
- Use plain text fields for stage, mood, and topic values in this iteration instead of Postgres enums, lookup tables, or check constraints.
- Do not add denormalized thread ordering or unread/unhandled fields yet.
- Do not overbuild the pending-message outbox fields yet.

## Acceptance Criteria

- A refined task defines the target tables, relationships, keys, indexes, and lifecycle fields for unified inbox data.
- The implementation removes the existing todo boilerplate migration, metadata, SQL helper, seed data, and GraphQL table tracking files where they conflict with the unified inbox schema.
- Generated migration(s) can be applied by the existing Hasura migration workflow.
- Generated seed data is enough to display a realistic unified inbox with populated conversations and some unhandled first-message threads.

## Dos

- Use Hasura/PostgreSQL-friendly SQL migrations.
- Use stable three-letter IDs where requested so static seeds are easy to read.
- Use `ui-avatars.com` URLs for tenant, user, and guest icons.
- Search for suitable real channel names and logo URLs, preferably channels that Hostaway integrates with.

## Don'ts

- Do not design the frontend inbox experience in this task unless explicitly added later.
- Do not introduce the removed todo model into replacement migrations, metadata, or seeds.
